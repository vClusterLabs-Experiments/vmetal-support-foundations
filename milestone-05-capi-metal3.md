# Milestone 5: Cluster API + Metal3

**Goal:** Replace every manual step in M1–M4 with declarative Kubernetes resources. A `BareMetalHost` CR points at the BMC. A `Cluster` + `KubeadmControlPlane` CR pair describes the desired cluster. Reconciliation loops do the work. You stop typing `curl` and `kubeadm init` and start typing `kubectl apply`, and the cloud-init payload you wrote by hand in M3 is now generated by a controller, delivered to the host as a ConfigDrive partition by Ironic, and applied by cloud-init at first boot exactly the same way.

**The single most important takeaway:** the *delivery mechanism* changes (NoCloud-over-HTTP → Metal3 BMH `userData` Secret → Ironic ConfigDrive). The *payload shape*, cloud-init cloud-config, does not. Read the M3 user-data and the Secret CABPK produces side-by-side. They are the same artifact in different envelopes.

## What you're actually learning

- **Declarative bare-metal management.** The operator pattern applied to physical hardware.
- **The reconciliation loop on hardware.** What happens when desired state and observed state diverge for a physical thing that takes 10 minutes to power-cycle.
- **The CAPI architecture.** Management cluster vs. workload cluster, infrastructure providers vs. bootstrap providers vs. control-plane providers.
- **CABPK**, Cluster API Bootstrap Provider Kubeadm. Its job is to *generate cloud-init user-data* from a `KubeadmConfig` (or `KubeadmConfigTemplate`) and stash it in a Kubernetes Secret. Read the Secret. Trace it back to what you wrote by hand in M3. Recognize them as the same cloud-config document.
- **Metal3's user-data delivery path.** Secret → `BareMetalHost.spec.userData` → Ironic → ConfigDrive partition on the deployed disk → cloud-init's ConfigDrive datasource → first-boot `runcmd` runs `kubeadm init` / `kubeadm join`.
- **Where vmetal fits.** vmetal is in this lineage. Understanding CAPI + Metal3 + CABPK means you can read vmetal's CRDs in M6 and immediately know what each one is for.

## Anchor question

> The cloud-init payload you wrote by hand in M3 still runs at first boot, but you didn't write it this time. Who wrote it, where does it live in Kubernetes, and how did it reach the disk?

Answer skeleton: a **CABPK controller** watching `KubeadmConfig` objects renders cloud-config YAML and writes a Secret. The contract is fixed: data key is `value`, and for `format: cloud-config` (the Ubuntu/cloud-init path this curriculum uses) the payload is gzip-compressed before being base64-encoded into the Secret. A **Metal3Machine controller** binds that Secret reference into the `BareMetalHost.spec.userData` field. The **Bare Metal Operator** instructs **Ironic** to deploy the host: Ironic powers it on via Redfish, the **Ironic Python Agent (IPA)** boots from RAM, writes the OS image to disk, *and writes a ConfigDrive partition (label `config-2`) containing the user-data*. The host reboots; cloud-init's ConfigDrive datasource reads the partition; first-boot `runcmd` runs `kubeadm init`/`join`. Same payload as M3, different mailman.

**Read first:** [cloud-init: ConfigDrive datasource](https://cloudinit.readthedocs.io/en/latest/reference/datasources/configdrive.html), the OpenStack-style `config-2` seed format Ironic materializes on the deployed disk, and the file paths cloud-init reads on first boot.

## Stack

```text
Lima VM
├── kind cluster                              <-- the management cluster
│   ├── CAPI core controllers
│   ├── CAPBK (Cluster API Bootstrap Provider Kubeadm)
│   ├── KCP (KubeadmControlPlane provider)
│   ├── CAPM3 (Cluster API Provider Metal3) + Bare Metal Operator
│   ├── Ironic (bare-metal provisioning service) + IPA ramdisk
│   └── BareMetalHost CRs                      <-- one per fake node
├── libvirt + QEMU                            <-- 2-3 fake nodes
├── network services                          <-- dnsmasq + HTTP from M2
└── sushy-tools                               <-- the BMCs
```

The Lima VM hosts a Kubernetes management cluster (in kind/k3s) whose only job is to orchestrate the bare-metal nodes. The workload cluster you'll provision is reconstituted *by* the management cluster; your M4 admin `kubeconfig` is throwaway from this milestone forward, CAPI mints a fresh one as part of the declarative bootstrap.

> **Linkback to inventory (M3.5):** the `BareMetalHost` CRs you'll write here are exactly "strategy 1: manual inventory" from M3.5, with the same fields (BMC URL, credentials, boot MAC). If those feel hazy, re-skim M3.5's strategy table before you start.
>
> **Linkback to M3:** keep your `seed/user-data` from M3 open in a second pane. You'll be diffing it against the Secret CABPK generates.

## Success criteria

1. **Management cluster running** with CAPI core + CABPK + KCP + Metal3 provider + Bare Metal Operator + Ironic installed.
2. **Three `BareMetalHost` CRs registered**, each pointing at sushy-tools BMC URLs and credentials. Watch one go through the BMH state machine: `registering → inspecting → available`. Observe Ironic driving Redfish under the hood (compare against your M1 manual `curl`s).
3. **Apply a `Cluster` + `KubeadmControlPlane` + `MachineDeployment` (workers) + `Metal3MachineTemplate`s + `KubeadmConfigTemplate` (workers)** and watch the management cluster provision a workload cluster automatically across your three fake nodes. The first CP node gets full `kubeadm init`; later workers get `kubeadm join`.
   **Reference:** [CAPI: Kubeadm-based control plane](https://cluster-api.sigs.k8s.io/developer/architecture/controllers/control-plane.html), the controller that drives `kubeadm init` and `kubeadm join --control-plane` in this flow.
4. **Follow the user-data, the central exercise.** For one of the provisioned `Machine`s, walk the chain end-to-end with `kubectl`:

   ```bash
   # Find the KubeadmConfig that owns this Machine's bootstrap data.
   kubectl -n <ns> get machines
   kubectl -n <ns> get kubeadmconfig

   # Find the Secret CABPK wrote.
   kubectl -n <ns> get kubeadmconfig <kc> -o jsonpath='{.status.dataSecretName}'

   # Decode the Secret. For format=cloud-config (this curriculum's path),
   # CABPK gzip-compresses the payload before base64-encoding it into the Secret.
   kubectl -n <ns> get secret <machine>-bootstrap-data \
     -o jsonpath='{.data.value}' | base64 -d | gunzip
   # (Other bootstrap providers, e.g. ignition for Flatcar/Talos, skip the gzip step;
   # not in scope for this curriculum.)

   # Confirm Metal3 wired the Secret into the BareMetalHost.
   kubectl -n <ns> get bmh <name> -o yaml | grep -A 2 userData
   ```

   Now diff the decoded payload against your M3 `seed/user-data`. Identify the lines CABPK adds (kubeadm config files, the `kubeadm init` or `kubeadm join` invocation) and the lines that are unchanged from your hand-written version (kernel modules, sysctls, `swapoff`, the containerd dance, the apt repo install). The cloud-init half is *the same artifact*. Write this diff up, it's the artifact that lands the milestone.

5. **Delete the workload cluster.** Watch the BareMetalHosts deprovision and return to `available`. Re-apply the same manifests; watch CAPI reprovision from scratch with new tokens, fresh certs, a freshly-generated bootstrap Secret.
6. **Reconciliation test.** While a node is `provisioned`, manually power it off via direct `virsh destroy`. Watch the controllers notice and react. What does Metal3 do? What does the `KubeadmControlPlane` controller do? What does the `MachineDeployment` controller do? Each layer reconciles its own desired state.
7. **Image + userData inspection.** `kubectl get bmh <name> -o yaml` and identify each of: `spec.image.url`, `spec.image.checksum`, `spec.image.checksumType`, `spec.image.format`, `spec.userData.{name,namespace}`, `spec.networkData`, `spec.metaData`. Trace the image URL back to whoever served it (M7 will have you build the image yourself).

## The "follow the user-data" trace, drawn explicitly

This is the diagram you should be able to draw from memory by the end of the milestone:

```text
KubeadmConfigTemplate (worker)            KubeadmControlPlane (CP)
        │                                        │
        ▼ (per-Machine generation)               ▼ (per-replica generation)
KubeadmConfig                            KubeadmConfig
        │                                        │
        ▼ (CABPK renders cloud-init)             ▼ (CABPK renders cloud-init)
Secret <machine>-bootstrap-data          Secret <machine>-bootstrap-data
.data.value: base64(gzip(#cloud-config…))
        │                                        │
        ▼ (CAPM3 wires reference)                ▼
BareMetalHost.spec.userData ──────────────►  same shape
BareMetalHost.spec.image.url     (raw OS image, M7 builds this)
        │
        ▼ (BMO ↔ Ironic)
Ironic powers on host via Redfish
Ironic Python Agent (IPA) boots from RAM, calls back to Ironic
IPA writes raw image to local disk
IPA writes ConfigDrive partition (label config-2) with user-data
Ironic sets boot device to disk; reboot
        │
        ▼ (deployed Ubuntu, first boot)
cloud-init ds-identify picks ConfigDrive
cloud-init reads /openstack/latest/user_data
runcmd: kubeadm init  (or kubeadm join)
Node Ready
```

Same payload shape, different envelope. The lesson lives in the comparison.

## Conceptual questions

Each question has a **Read first** pointer to a source that teaches the underlying concept.

### 1. Why split Metal3 into CAPI provider + BMO + Ironic + IPA?

What does each component own, and why this division of labor? Where does CAPM3 stop and BMO start? Where does BMO stop and Ironic start? Where does Ironic stop and IPA start? Why is IPA a separate ramdisk that runs on the host being provisioned, rather than just an Ironic driver action?

**Read first:**
- [Metal3: Bare Metal Operator](https://book.metal3.io/bmo/introduction.html), the K8s-side BareMetalHost API and controller responsibilities.
- [Metal3: Ironic in Metal3](https://book.metal3.io/ironic/introduction.html), describes Ironic's role inside Metal3 and the related services in the stack.
- [Ironic: Architecture](https://docs.openstack.org/ironic/latest/contributor/architecture.html), Ironic's service architecture, driver model, and the IPA ramdisk's role.

### 2. Management cluster dies: are workload nodes orphaned?

What state lives only on the management cluster, what lives on the workload cluster, and what would survive a complete loss of the management cluster? When you re-stand-up a management cluster from a backup, how do the existing nodes re-attach?

**Read first:**
- [CAPI book: Pivoting (`clusterctl move`)](https://cluster-api.sigs.k8s.io/clusterctl/commands/move.html), the *mechanism* for moving Cluster API objects between management clusters.
- [CAPI: Glossary, pivot, management cluster, workload cluster](https://cluster-api.sigs.k8s.io/reference/glossary), names the *scenario*. Pair with the `move` command.
- [CAPI book: Concepts](https://cluster-api.sigs.k8s.io/user/concepts), the relationship between management clusters, workload clusters, and Cluster API objects.

### 3. Reconciliation cost on physical hardware

A software controller can re-reconcile every 10 seconds; a bare-metal controller cannot, power cycles are slow, image writes are long, BMC operations are expensive. What does this force the controller design to look like? Where does Metal3's `BareMetalHost` state machine make the cost explicit, and how do CAPI's Machine and KubeadmControlPlane controllers handle the long tail of "this won't be ready for 10 more minutes"?

**Read first:**
- [Kubebuilder book: Controllers](https://book.kubebuilder.io/cronjob-tutorial/controller-overview), the reconcile loop pattern, requeue semantics, retry/backoff behavior.
- [Metal3: Host State Machine](https://book.metal3.io/bmo/state_machine), the explicit state machine Metal3 uses for long-running host transitions.

### 4. CABPK and the user-data delivery boundary

CABPK generates cloud-init user-data. Metal3 delivers it. Why are those two responsibilities split? What's in the contract between them, what shape does the bootstrap Secret have to be in for Metal3 to consume it correctly? What changes about the Secret if the workload OS image were Talos instead of Ubuntu? (Trick question, the answer ties to why the curriculum committed to Ubuntu/cloud-init.)

**Read first:**
- [Cluster API: Bootstrap Provider Kubeadm](https://cluster-api.sigs.k8s.io/tasks/bootstrap/kubeadm-bootstrap.html), how `KubeadmConfig` becomes cloud-init user-data.
- [Metal3: Provisioning](https://book.metal3.io/bmo/provisioning), the BMH `userData` Secret consumption side.

### 5. Image management

Where does the OS image served to the node actually come from? Who builds it, where is it stored, and how does Metal3 reference it? What's `spec.image.checksumType` for, and what failure mode does it prevent? Why does Metal3 prefer raw images over qcow2 for predictable deploy time?

**Read first:**
- [Metal3: Provisioning and Deprovisioning](https://book.metal3.io/bmo/provisioning), `image.url`, checksum, format, deprovision/reprovision behavior.
- [Metal3: Ironic in Metal3](https://book.metal3.io/ironic/introduction.html), Ironic's role in fetching, validating, and writing the image.

### 6. Locking: who owns a physical node?

A `BareMetalHost` is a finite resource. Two `Machine`s cannot claim it. Where does the lock live, who sets it, who clears it, and what happens during a deprovision-then-reprovision cycle?

**Read first:**
- [Metal3: Host State Machine](https://book.metal3.io/bmo/state_machine), the operational states where a host is available, provisioned, or being reclaimed; `consumerRef` semantics.
- [CAPI: Machine controller](https://cluster-api.sigs.k8s.io/developer/core/controllers/machine), the Machine ↔ infrastructure object binding contract.

### 7. vmetal mapping (preview for M6)

Read vmetal's docs and identify which of its CRDs map to CAPI/Metal3 primitives, which wrap them, and which are net-new. Pay specific attention to anything that looks like it would *modify the cloud-init payload* on the way to the host (Tenant Cluster registration scripts, GPU-specific bootstrapping). That's where vmetal's value-add lives, and it's where M6's runcmd-customization question lands.

**Read first:**
- [vmetal Architecture](https://vmetal.ai/docs/architecture/) and [vmetal Configuration](https://vmetal.ai/docs/configuration/), start here before reading installed CRDs.
- [CAPI: Developing providers](https://cluster-api.sigs.k8s.io/developer/providers/overview) and [InfraMachine contract](https://cluster-api.sigs.k8s.io/developer/providers/contracts/infra-machine), the contract every infrastructure provider has to satisfy is the lens for reading vmetal's CRDs.
